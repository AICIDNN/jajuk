Developer guide

Welcome to new Jajuk developers ! Have fun.

Natural Language

Jajuk code and comments must be only in English natural language.

Mailing List / Forum

It is mandatory to subscribe to Jajuk developer mailing list ( jajuk-developers@lists.sourceforge.net ) at http://lists.sourceforge.net/lists/listinfo/jajuk-developers as this list is the main communication stream. You may also use and monitor the public Open Discussion forum (http://sourceforge.net/forum/forum.php?forum_id=315360)

Developers activity and list can be found at http://jajuk.sourceforge.net/developers.html

Voting system

Jajuk uses Apache Group voting system (http://httpd.apache.org/dev/guidelines.html) to take decisions.

IDE

We strongly advice to use the Eclipse platform to develop. All following documentation (CVS...) deals with Eclipse only.

CVS rules

  • Make sure the whole project yet compiles and is in an acceptable stability state before committing something. if you feel your work not stable enough, keep it on your local workspace for the moment.

  • During a release process, please follow freezes (see release process part).

  • You should work most of the time in the head branch for minor changes.

  • If you start to code a big feature (like P2P) with a lot of impacts on the project and potentially dangerous for current release, create your own branch for this feature and merge your work into the head when you feel ready and when no release is planned for some weeks.

  • All bugs should be double-fixed in the maintenance branch of last release + on the head (see version rules bellow).

Version rules

  • Jajuk releases (from 1.0) follow this version scheme: <major>.<minor>.<fix>. Example: 1.1.4 : Fourth fix release of second minor release of second major release.
  • When a new minor release starts, we create a maintenance branch used after releasing for fix releases. Example: when starting 1.1 release in the head, we create a maintenance_1_1 branch. All important bugs found against minor version after releasing are double-fixed in the maintenance release and in the head of next minor release. For example, an issue 123456 discovered on 1.1 is fixed in the maintenance_1_1 branch AND in the head (future 1.2 release). When we decide that the maintenance release is ready (when reaching a given number of bugs, time or if we need an immediate release), we create and publish a 1.1.x version.

Eclipse configuration

  • In preferences / "Java/Code generation/Code and comments" or "Java/Code Style/Code templates", please import scripts/code_generation.xml file. This is a template for new java sources.

  • When you launch Jajuk from eclipse, you have to specify '-debug' as program argument so classes in directories takes images from local jajuk.jar file and doesn't look for images with an url from inside the jar.

  • You can also use the '-notaskbar' as program argument to hide program in task bar and keep only tray.

Use -ide, -test and -notaskbar arguments:

  • -ide argument tells jajuk that it is executing inside an IDE (probably Eclipse) . This changes the way of getting resources (images, doc...).
  • -test argument tells jajuk to use the ~/.jajuk-debug repository and not the ~/.jajuk, reserved for stable releases (this way a jajuk developer can use a stable release of jajuk when working on a new release) .
  • -notaskbar (optional) : tells jajuk to hide from task bar when reduced as tray.

  • Only use UTF-8 for your source. Use preference screen to set it:

Check your .properties file uses correct encoding (especially under Eclipse 3.0 final) by selecting file/properties/info. If the file is in ISO8859-1, set UTF-8 instead.

ant scripts and i18n

Read this if you are a developer planning to make also some translation ! You shouldn't have to use any provided ant script when under Eclipse environment. Two ant scripts are found in the tree:

  • scripts/build.xml : this is the stored build file packaged with the distribution and used in command line to recompile Jajuk using a standard installation. Do not use it under Eclipse.

  • scripts/build-dist.xml : this is the ant script used by the packager to generate Jajuk distribution. You shouldn't have to use it.


Release process

The Jajuk release process usually takes from 2 to 4 weeks.

  • The release process is announced with a message in the developer mailing list (make sure you subscribed to this list). Obviously, all developments are frozen in the repository. English labels are frozen to allow translators to begin there work.

  • Translators can complete there translations during the entire process but the soonest would be the best to allow testers to use up-to-date translated releases. Note that translators can translate current releases even outside release process periods. Translator guide is available here.

  • Testers qualify and developers fixes bugs discovered during this period. We try to release a release candidate (rc) every 2 or 3 days during this period. Note these bugs and release are internal only and you should use to public bug tracker to report them but use developer list instead. All these rc are internal only and should not be distributed to public. Testers can base their works on this qualification form. A mail is send on developer list at each rc publication.

  • When we estimate that the last rc is stable enough, the packager releases it on SourceForge as stable release. The communication delegate issues the new in Freshmeat and other web sites.



Code conventions

We mainly follow Sun Java conventions with special code standards :

String externalization

  • All application labels have to be externalized (see translator guide) as <Class name>.<number from 0> .
  • For common labels like OK, Cancel... we use standard externalized labels (see jajuk.properties)
  • Do not fill holes in an existing class label set but always use a new number for new strings because removed labels can still be present in out-of-date langpacks.

Code structure

You can import the scripts/formatter.xml description file in Window->Preferences. Then use Right click->Source->Format to format your code according to Jajuk standards.

public void <(){
  if (size == 0) {
    try {
      size = (long) inStream.available();
    } catch (IOException e) {
       e.printStackTrace();
    }
  }
  else if (size == currentSize) {
    ++size;
  }
  else {
    --size;
  }
}

Variables and attributes names

If needed, begin variable name by the first letters of the type so the variable is auto-described:

String sFoo StringBuffer sbFoo Track track;

Interfaces

Begin interfaces with a upper ' I '. The 'i' is not taken into account for variables names:

IPerspectiveManager pm;

XML

  • Avoid to use PCDATA values, use attributes instead : it's faster to parse and much easier to code.

  • Use single quotes, not double quotes.

Comments / Javadoc

  • Please correctly comment attributes and methods with javadoc norms ( /** comments */ ).

  • Please remove dead code and move it to some personal junk classes if you want to keep it.


Project Rules

Becoming a new developer

  • A new developer will be added in the project only after having written significant code and having accepted the Team rules (see below). 
  • This code can be a feature implementation chosen from task list ( http://sourceforge.net/pm/?group_id=91412 ) or from its choice. 

Team rules

We ask members have to follow these basic rules :

  • Members have to be SourceForge member.

  • Members have to communicate in English.

  • Members have to subscribe on developer mailing list. This is the main communication stream so people don't receiving messages can't be considerate as project members. Unsubscribing from this list will be understood as a mean to leave the project.

  • Members should give minimum planning and actions on the project so we can plan releases.

  • Members have to follow CVS rules.

Leaving project

  • Any member can leave the project, please tell if you give up or have no more time for it: it will simpler to know who is really working on project and who doesn't.
  • Members may be removed from the project if we don't heard about them after 3 mouths.
  • Any member can leave project and come back later but we prefer to manage a list of active members only. 
  • Left members will keep their credit in Jajuk About View.
